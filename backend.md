# Architecture Overview (Realtime Sync with Firebase)  
Modern chess apps use persistent connections and real-time data sync. Firebase’s Realtime Database (RTDB) provides built-in low-latency updates and presence detection, while Cloud Firestore offers richer queries and global scaling. A common design is to use **RTDB for live/ephemeral data** (match queues, in-game state, presence) and **Firestore for persistent storage** (user profiles, completed games, leaderboards). Clients maintain websocket listeners on relevant nodes (e.g. game states or chat) so that updates (moves, clock changes, opponent online status) propagate instantly.

## Matchmaking Design (Time Controls and Queues)  
Create **separate queues for each time control** (e.g. 5|0, 10|0) in the RTDB, such as `/match_queue/5_0/{playerId}`. When a player requests a match, write their UID to the queue node for that time control. A Cloud Function (triggered on new queue entries) then checks if enough players (usually 2) are waiting. If so, the function atomically pairs players and creates a new game record (for example a Firestore `games/{gameId}` document) with both player IDs, time control, and initial clock times. The function should use a database transaction or batch write to remove players from the queue and update their entries concurrently, preventing race conditions. For instance, the function could also write `players/{uid}/gameId = gameId` so each client sees its `gameId` and knows to navigate to the game. Optionally, you can factor player rating (ELO) into the matching logic by checking ratings before pairing, but a simple FIFO or random match queue often suffices for casual play.

## Data Models (Firestore / Realtime DB Structure)  
Use both databases in complementary ways. For example:

- **Users** (`Firestore: users/{uid}`): Store profile data (username, country, rating, coin balance, badges, achievements). Include fields like `rating`, `coins`, and historical stats. A subcollection or map can track achievements and badges (e.g. wins in each game mode). Also keep a `last_active` timestamp if enforcing inactivity logouts.  
- **Matchmaking Queues** (`RTDB: /match_queue/{timeControl}/{playerId}`): Lists of UIDs waiting for each time control. A Cloud Function monitors these for pairing.  
- **Games** (`Firestore: games/{gameId}`): Each game document holds metadata (time control, player1/2 UIDs, start time, outcome, winner UID, and final clock values). Store the game’s status (ongoing/completed) and any other persistent stats. You can also mark whether players are currently “in-game” by including e.g. a `status` field on each player’s user doc or in the game doc.  
- **Moves** (`Firestore` or `RTDB`): Maintain a chronological list of moves. One approach is a subcollection `games/{gameId}/moves` in Firestore, where each doc is one move (fields: SAN/UCI notation, timestamp, elapsed clock). Alternatively, use RTDB path `/games/{gameId}/moves/{moveNumber}` for faster realtime syncing. Store timestamps or move clocks to allow replay and lag-compensation. This enables back-and-forth navigation by querying the moves array up to a given index.  
- **Clocks and Timestamps**: Do not write every tick. Instead, store initial start time and remaining time on each move. Clients run their own timers and update the DB only when a move is made (or if a player flags out/disconnects). Record each move’s timestamp so the server can recompute elapsed times if needed (useful for Chess.com–style “lag forgiveness”).  
- **Chat**:  
  - *In-Game Chat* (`RTDB: /chats/{gameId}/{msgId}`): Real-time messages during a match. These are ephemeral: use a Cloud Function to delete or disable the chat subnode when the game ends (for example, by clearing `/chats/{gameId}` in RTDB).  
  - *Out-of-Game Chat* (`Firestore: global_chats` or per-room): Messages in lobbies or friend chats. Store each message with a timestamp. Use a TTL policy or scheduled Cloud Function to delete documents older than 1 day. (Firestore now supports TTL fields that can auto-expire documents roughly 24 hours after the timestamp, reducing manual cleanup.)  
- **Presence/Status** (`RTDB: /status/{uid}`): Use Realtime Database to track online/offline state. Each client writes its status with an `onDisconnect()` handler: for example, set `/status/uid = {state: 'online', last_changed: serverTimestamp}` when connected, and automatically set it to `'offline'` on disconnect. Optionally mirror this into Firestore (via a Cloud Function) if you want to query user statuses in Firestore. This lets the app display “online friends” or count active users by querying how many `/status/` entries have `state: "online"`.  
- **Transactions/Coins** (`Firestore: users/{uid}/transactions`): To handle the coin system, use Firestore transactions (or a Cloud Function) to atomically deduct entry stakes and award winnings. For example, when creating a game, a transaction can check each player’s coin balance and subtract the entry coins; after the game, another transaction credits the winner and logs a record in `users/{uid}/transactions/{txId}` (fields: gameId, amount, balance_after). Always update balances and write transaction records in a secure transaction to prevent cheating.  

Shallow reads and targeted listeners are key: clients should only listen to the specific paths they need (e.g. `/games/{currentGameId}`, `/match_queue/5_0` or `/status/{friendUid}`), not entire collections. This minimizes bandwidth and costs.

## Timer System  
Implement chess clocks on the client for smooth UI updates. When a game starts, store the start timestamp and initial time control in the game record. Each time a player makes a move, record the move along with the current timestamp and remaining time. The opponent’s app will receive the move via Firebase listener and can update its local clock accordingly. This way, clocks are synchronized by comparing timestamps rather than constant DB writes. In practice, only write clock updates on moves or game end. This mimics Chess.com’s handling of latency: small network delays are absorbed by using server timestamps, and “lag compensation” can be implemented by ignoring differences below a threshold. Avoid writing clock ticks each second; instead rely on local countdown and occasional sync via move timestamps.

## Move Logging and Navigation  
Every move should be stored in order, enabling users to navigate forward/backward through the game. For example, use a `games/{gameId}/moves` subcollection (or equivalent RTDB list) where each document includes `{ san: 'e4', from: 'e2', to: 'e4', timestamp: ..., clockWhite: 600, clockBlack: 600 }`. Clients can subscribe to this subcollection (or read it on-demand) to display move history. When a user presses “back” or “next”, simply load the move with that index. For efficiency, consider fetching moves in pages (e.g. first 10 moves, then more on demand) or caching the moves array in memory. Security rules or a Cloud Function can validate each move entry if needed (e.g. disallow illegal moves), following a pattern where the client writes a “proposed move” and a Cloud Function computes the resulting state before writing it officially. This ensures game integrity even if clients are untrusted.

## Player Presence and Reconnection  
Use Firebase’s presence system to detect if players disconnect. As noted, the client writes its online status to `/status/{uid}`. If a player loses connection, `onDisconnect()` will mark them offline. On reconnect, the client should check if it has a `gameId` (stored locally or in `/players/{uid}`) and re-join that game. The UI can then resume from the last known game state (with moves and clocks) from Firestore or RTDB. If a player disconnects for too long (e.g. timeout), the server can declare a forfeit. To implement “logout after 5 minutes of inactivity,” simply track a last-action timestamp (e.g. update a field on user doc) and either have client-side code sign the user out after idle, or run a periodic Cloud Function that signs out or flags users whose last action was >5 minutes ago. The **active user count** can be derived by counting `/status/{uid}` entries with `state: 'online'`.

## Chat and Social Features  
**Spectating:** Allow a user to view a friend’s game in realtime by giving them read access to the friend’s `games/{gameId}` data and chat. They can subscribe to the game’s moves and in-game chat in the same way a player does, but with rules enforced so they cannot write moves or influence the game. This can be handled by security rules (e.g. only players can write to `moves` nodes).

**In-Game Chat:** Implement chat messages as part of each game’s state (e.g. `RTDB /chats/{gameId}/{msgId}`). Use RTDB for low-latency messaging. When the match ends, trigger a Cloud Function to delete or archive `/chats/{gameId}` so old in-game messages are removed.

**Out-of-Game Chat:** For lobby or friend chat, store messages in Firestore with timestamps. Configure a TTL policy or scheduled function to delete messages older than 1 day. For example, mark each message with `expiresAt = timestamp + 1 day` and let Firestore automatically purge it (deletion typically occurs within 24 hours after expiration). This ensures chat history is ephemeral as required.

## Deployment & Optimization  
Host the web frontend on **Firebase Hosting** for fast CDN delivery. Use **Cloud Functions** for server logic: matchmaking triggers, game start/end processing, chat cleanup, and coin transactions. Optimize database usage by using **targeted listeners**: each client only subscribes to its own game document or relevant queue/status paths. Use **shallow reads**: store moves as subcollections so fetching a game doc doesn’t automatically fetch hundreds of moves unless requested. Beware of write costs: for example, avoid writing the clock value every second (clients keep the clock locally and only update the DB on move or game end). Use batched writes or Firestore transactions for atomic updates (e.g. decrement two players’ coin balances in one transaction). 

Enforce security with **Firestore/RTDB Security Rules**. For example, only allow writing a move if it’s the current player’s turn and the move is well-formed; only allow reading a game’s moves/chat if the user is a participant or allowed to spectate. For complex game logic or financial updates (coin transfers), run Cloud Functions on data writes to validate rules and update state. This separation ensures only legitimate moves/transactions are applied.

## Summary  
By combining RTDB and Firestore, you can build a scalable, real-time chess backend. Use RTDB for low-latency features (match queues, presence, live moves/chat) and Firestore for persistent data (user profiles, match history, analytics). Employ Cloud Functions for matchmaking logic and transactional updates. Store moves and timestamps to support replay and “lag forgiveness,” run clocks on the client, and clean up ephemeral data with TTL or triggers. With careful data modelling, targeted listeners, and security rules, this design minimizes continuous reads/writes and leverages Firebase best practices for a smooth, chess.com–style multiplayer experience.
